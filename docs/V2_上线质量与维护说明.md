# V2 上线质量与维护说明

## V2 功能说明

V2 当前已经覆盖这些内部工具能力：

- 风格广场：30 个风格包，支持搜索、优先级、类别、标签筛选。
- 模板库：40 个模板，支持平台、类型、风格包、场景、状态、代码链接筛选。
- 模板交付页：服务业务、设计、开发和 AI 工作流，包含预览状态、页面结构、组件依赖、设计交付、开发交付和 Prompt。
- 组件文档：读取组件 JSON，展示分类、状态、变体、Token、Storybook、Code、使用模板。
- 设计维护：包含 `/design-guide` 和 `/admin/design-handoff`。
- 开发者入口：包含 `/developer`，说明如何从模板进入真实开发。
- AI 推荐：结构化规则推荐，不接真实大模型 API。
- 维护后台：前端可编辑、校验、导出 JSON，但暂不写入数据库。

## 数据维护说明

核心数据文件在 `data/`：

- `styles.json`：风格包。
- `templates.json`：页面模板。
- `components.json`：组件目录。
- `prompts.json`：Prompt。
- `tokens.json`：Token。
- `changelog.json`：更新记录。
- `release-notes.json`：版本说明。

维护要求：

- ID 必须唯一，并且适合作为 URL。
- 模板必须绑定存在的 `style_pack_id`。
- 模板必须绑定存在的 `ai_prompt_id`。
- 模板组件可以使用组件库名称，也可以使用已认可的业务别名。
- Figma、Storybook、Code 链接必须是有效 URL。
- 模板必须维护设计验收项 `design_acceptance`。

每次改数据后运行：

```bash
npm run validate:data
```

## 角色权限说明

第一版不接真实登录，但已经预留角色模型：

- `viewer`：只能查看风格、模板、组件和推荐结果。
- `designer`：可维护风格包、模板预览、Token、Figma 链接和设计验收项。
- `developer`：可维护组件、Storybook、代码链接和开发交付说明。
- `admin`：可审核、发布、废弃数据，并负责上线质量把关。

角色定义在：

```text
lib/roles.ts
```

后续接真实权限时，可以把这些角色和权限迁移到数据库。

## 上线质量检查

上线前建议依次运行：

```bash
npm run quality:check
npm run build
```

`quality:check` 会检查：

- 数据必填字段。
- ID 是否重复。
- 风格、模板、组件、Prompt 绑定关系。
- 关键页面文件是否存在。
- 动态路由数据是否完整。
- 空状态、错误状态、复制反馈、筛选清空等质量标记是否存在。

`npm run build` 用于确认项目可以部署到 Vercel。

## 部署说明

第一版没有数据库、没有真实 AI API，也没有私密环境变量。

部署到 Vercel 时：

- Framework Preset 选择 Next.js。
- Build Command 使用 `npm run build`。
- Environment Variables 暂时不需要配置。
- JSON 数据会在构建时一起打包。

## 后续接真实数据库说明

建议数据库表保持当前 JSON 的语义：

- `styles`
- `templates`
- `components`
- `prompts`
- `tokens`
- `changelog`
- `release_notes`
- `users`
- `roles`
- `permissions`

优先迁移顺序：

1. 先把 `/admin` 的导出 JSON 改为保存到数据库。
2. 增加登录和角色权限。
3. 给 draft、reviewing、approved、deprecated 状态接审核流。
4. 接 Figma、Storybook 和代码仓库真实同步。
5. 接真实 AI API，并让 AI 基于数据库 metadata 检索推荐。

## V3 暂不做

- 真实登录和权限。
- 后台写入数据库。
- Figma API 自动同步。
- Storybook iframe 真嵌入。
- 真实 npm 组件库发布。
- AI 大模型实时推荐。
- 审核流通知、评论、多人协作。
